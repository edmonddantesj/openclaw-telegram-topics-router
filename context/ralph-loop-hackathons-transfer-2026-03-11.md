## Transfer intent
Hackathons should not be treated only as one-off event participation.
Inside Ralph Loop, hackathons becomes an **evidence-first repeated scouting / benchmarking / intelligence loop**.

Short-term goal:
- capture good hackathon opportunities fast,
- assemble submission-ready package inputs,
- surface deadlines/blockers/proof early.

Long-term goal:
- accumulate reusable proof-first AI agent infrastructure assets,
- turn repeated hackathon work into reusable playbooks, templates, demos, and receipts.

## Source-topic truth retained in hackathons
These should remain primarily in `hackathons` topic/playbook:
- event-specific context,

- final external submission gating.

## What moves into Ralph Loop
The following work is Ralph-loop-shaped and should be operated from Topic 68:
1. deadline sweep / repeated scouting cadence
2. benchmark extraction from prior winners or strong repos
3. signal collection (themes, judges, sponsor priorities, infra requirements)
4. synthesis into reusable task cards / templates
5. routing repeated package-building chores into smaller execution units

## Recurring task shapes for Ralph Loop
### RL-HACK-01 Scout
- scan new/relevant hackathons
- record deadline / theme / eligibility / prize / infra fit
- produce proof links

### RL-HACK-02 Deadline Sweep
- rank upcoming opportunities by urgency and fit
- flag blockers (team, assets, submission requirements, missing demo)

### RL-HACK-03 Benchmark
- inspect winning or high-signal projects/repos
- extract reusable patterns: demo shape, judging hook, stack, proof style

### RL-HACK-04 Signal
- summarize what the market/judges currently reward
- classify by near-term tactical vs long-term infrastructure value

### RL-HACK-05 Synthesis
- convert findings into templates, checklists, or small tasks
- push event-specific execution back to `hackathons` when needed

## Boundaries / gates
- Internal scouting / benchmarking / packaging prep: L1/L2
- External submission / public posting / irreversible commitments: L3 gate retained
