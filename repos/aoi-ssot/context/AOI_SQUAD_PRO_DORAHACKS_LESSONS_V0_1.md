# AOI Squad Pro 톤 레슨: Tangerine 프로젝트 비교 정리 (v0.1)

## 1) 채택할 만한 포인트 (3개)

### 1. 역할 분업의 정량화
- 오케스트레이터 + 전담 역할 구조를 **입력/판단/실행/검증** 경로로 명시.
- AOI 적용안: 현재 `Planner-Researcher-Builder-Reviewer-Operator` 흐름에서 각 단계 산출물을 필수로 포맷 통일.

### 2. 합의 규칙을 데이터화
- Tangerine은 합의 임계값(예: 신뢰도/리스크 조건) 기반으로 실행 여부를 gate함.
- AOI 적용안: task-type별 허용 조건 + 승인 워크플로우(approve/reject) + proof-first 규칙 결합.

### 3. 실행 로그의 검증 가능성
- 오퍼레이션 히스토리, 로그 경로, 트랜잭션 추적 가시성을 강조.
- AOI 적용안: `/tmp/aoi_squad_pro_run_<ts>/` 내부에 input_digest, sha256, proof.sha256를 기본 증빙 항목으로 고정.

## 2) 채택하지 말아야 할 포인트 (3개)

### 1. 과도한 온체인 자동 트레이딩 의존
- AOI는 현재 실무 오케스트레이션 성숙 단계이며, 실시간 온체인 거래 자동 실행은 리스크가 큼.
- 도입시: 보안·회수·감사 부담이 급격히 증가.

### 2. 복합 스택의 조기 고착
- Tangerine은 BSC/React/Wagmi/Wallet 등 다중 스택 의존도가 높음.
- AOI 초기 단계에선 과부하·통합 지연·유지보수 난도 증가 위험.

### 3. 트레이딩 중심 전략 과도 확장
- 우리 미션은 우선 `proof-first 협업 자동화`와 `approval governance` 안정화.
- 실시간 트레이딩을 먼저 붙이면 KPI가 산만해지고 감사/배포 일관성이 깨질 수 있음.

## 3) 경량 실험 2개 (다음 1주)

### 실험 A: `증빙 강건성` 스모크 테스트
- 목적: 제안/승인/적용 루틴에서 증빙 누락률 0% 달성
- 방식: 10건 샘플 과제에 대해 자동 체크
  - proof_dir 존재
  - input_digest 존재
  - sha256 존재
  - logs 경로 존재
- 성공 기준: 10건 모두 PASS

### 실험 B: `합의 게이트` 임계값 튜닝
- 목적: 불필요한 오탐지/반복 승인 절차를 줄이기
- 방식: task-type별 승인 규칙을 2주치 데이터로 오프라인 튜닝
- 핵심 지표: BLOCK 비율, NEEDS_SUPPORT 비율, 승인 소요시간(평균)

## 4) 요약 제안
- Tangerine은 참고 재료이지 AOI를 복제할 대상 아님.
- AOI는 지금은 **"실행성 + 증빙성 + 감사 가능성"**을 먼저 고정한 뒤, 유사 기능을 느슨하게 흡수.
- 운영 문장: `검증된 합의형 오케스트레이션` → AOI만의 차별화 축.
